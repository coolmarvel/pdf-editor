# ADR-0001: file-converter에서 PDF 편집 전용 프로젝트로 분리

## 상태

Accepted (2026-07-08, 사용자 결정)

## 맥락

`file-converter`(파일 변환기)는 PDF·이미지·DICOM 상호 변환 앱으로 시작했으나, 사용자의 요구가
pdfguru.com 수준의 **PDF 편집기**(페이지 관리, 텍스트 수정, 그리기, 스탬프, 서명, 주석, 링크…)로
이동했다. 변환기 UI(우측 옵션 사이드바 + 파일 큐) 위에 편집 기능을 계속 덧붙이면 `App.tsx`가 비대해지고
UI 구조 자체가 pdfguru 레이아웃(중앙 상단 툴바 중심)과 충돌한다. 사용자도 "PDF 전용으로 새로 파자"고 결정.

## 결정

1. `/home/jace/pdf-editor` 를 신설한다. 파일 변환기는 그대로 두고(변환·워터마크 용도) 여기서는 PDF 편집만.
2. 검증된 스택은 승계: electron-vite + React + TS + pdf.js + pdf-lib, NSIS/Wine 빌드, 창 복구 로직.
3. UI는 **MUI(@mui/material)** 로 재구축 — 사용자 요구("템플릿/컴포넌트 라이브러리, 모듈화·재사용").
4. 상태는 **zustand** 단일 스토어, 순수 로직은 `src/core` 로 분리해 node 테스트.
5. 문서 하네스(CLAUDE.md 부팅 프로토콜, session-log SSOT, 스크린샷 피드백 루프, 버전 정책 1.x.x,
   검증 후 인스톨러 자동 굽기)를 file-converter에서 그대로 이식.

## 근거

- 기존 프로젝트 확장 대안: 코드 재사용은 쉽지만 UI 정보구조가 근본적으로 달라 리팩터 비용 > 신설 비용.
- MUI vs antd: 데스크톱 번들 크기 부담이 없고, 아이콘 세트·테마 커스터마이즈가 pdfguru 스타일 재현에 충분.
  antd는 데이터-테이블 계열에 강점이나 이 앱은 툴바·캔버스 중심.

## 결과

- (+) pdfguru 레이아웃을 처음부터 올바른 구조(툴바 중심, 페이지 캔버스 스테이지)로 구현.
- (+) core 순수 로직 분리로 페이지 연산·히스토리가 테스트로 보호됨.
- (−) 파일 변환기와 브랜딩 자산(build/)이 당분간 중복 — PDF 전용 아이콘 교체는 todo P3.
- (−) 워터마크 등 일부 기능은 이식 전까지 두 앱에 나뉘어 있음 (todo P2).
