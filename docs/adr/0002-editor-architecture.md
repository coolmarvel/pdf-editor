# ADR-0002: 편집기 아키텍처 — 정규화 객체 모델 + 저장 시 평탄화

## 상태

Accepted (2026-07-08)

## 맥락

편집 객체(텍스트·선·도형·이미지·노트·링크)를 화면 표시, 페이지 회전, 재정렬, 최종 PDF 저장까지
일관되게 다뤄야 한다. 화면 배율(줌)과 내보내기 해상도가 다르고, 페이지는 소스 자체 회전(/Rotate)에
사용자 추가 회전이 얹힌다.

## 결정

1. **좌표는 전부 표시(뷰포트) 기준 정규화(0..1)** — x,y,w,h = 페이지 폭/높이 대비 비율.
   렌더러(`editor/draw.ts`) 하나를 라이브 오버레이와 내보내기가 공유해 화면=결과물 1:1을 보장.
2. **문서 = PageRef[] + objectsByPage** — 페이지 연산(삭제·복제·이동·회전·가져오기)은 소스 PDF를
   건드리지 않는 참조 배열 조작. 소스 바이트/pdf.js 프록시는 `pdf/docs.ts` 모듈 맵에 격리(스토어 밖).
3. **페이지 회전 시 객체 좌표도 함께 90° 변환**(`rotateObjectCW`) — 객체가 페이지 내용에 붙어 다닌다.
4. **저장은 평탄화 방식**: 새 PDFDocument에 `embedPage` + 회전 transform(4케이스)으로 페이지를 표시
   방향 그대로 얹고, 오버레이는 표시 공간에서 투명 PNG(≈180dpi)로 구워 페이지 전체에 drawImage.
   링크/노트는 굽지 않고 실제 PDF 주석(Link/Text Annot)으로 삽입.
5. **undo/redo는 스냅샷 스택**(pages+objects+sizes). 불변 갱신이라 얕은 보관으로 충분, 상한 100.

## 근거

- 평탄화 대안: 원본 페이지에 /Rotate·Annots를 직접 조작하면 회전 좌표계(4케이스×원본회전)와
  기존 주석 보존 문제가 폭증. embedPage 방식은 페이지 연산·빈 페이지·타 문서 가져오기를 동일 경로로 처리.
- 오버레이 PNG 대안: 객체를 pdf-lib 벡터(drawLine/drawText)로 재현하면 한글 폰트 임베드·blend 모드
  재현이 복잡. 캔버스 굽기는 "화면에서 보이는 그대로"를 보장(파일 변환기에서 검증된 방식).
- 트레이드오프: 원본 페이지의 기존 링크/북마크는 평탄화에서 사라짐, 오버레이는 비트맵(원본 텍스트는 벡터 유지).

## 결과

- (+) 도구 추가가 정형화됨: 객체 타입 → draw → 인터랙션 → (필요시) save. CLAUDE.md "코드 지도" 참조.
- (+) 회전·복제·가져오기 전부 순수 함수라 테스트 가능(`test/pages.test.ts`).
- (−) 저장물에서 원본 문서의 내부 링크 소실 — 필요해지면 주석 복사 로직 추가 (todo에 없음, 요구 시).
